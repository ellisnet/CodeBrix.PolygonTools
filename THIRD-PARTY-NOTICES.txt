CodeBrix.PolygonTools is a fully managed 2D polygon clipping and offsetting
library for .NET. It performs the four boolean set operations - intersection,
union, difference and exclusive-or - on arbitrary sets of polygons and open
paths, and inflates or deflates polygons with miter, round or square joins.

   MODIFICATIONS MADE
   ---------------------------------------------------------------------------

   The following changes were made to the upstream source during the port.
   None of them alters the clipping or offsetting algorithms.

   a) Type renames. Five identifiers containing "Clipper" were renamed so that
      the CodeBrix library does not carry the upstream product name in its
      live code:

          ClipperLib (namespace) -> CodeBrix.PolygonTools
          Clipper                -> PolyClip
          ClipperBase            -> PolyClipBase
          ClipperOffset          -> PolyClipOffset
          ClipperException       -> PolyClipException

      No other identifier was renamed. In particular ClipType, PolyType,
      PolyFillType, JoinType, EndType, IntPoint, IntRect, DoublePoint,
      PolyNode, PolyTree and all enum member names are unchanged from upstream.
      Occurrences of "Clipper" in comments - including the two "see comments in
      clipper.pas" provenance notes and the "translation of the Delphi Clipper
      library" line in the file header - were deliberately left intact.

   b) File split. The single 4,913-line clipper.cs was split into one file per
      type, organized into Models/, Enumerations/ and Internal/ sub-folders per
      CodeBrix family convention. Type bodies were dedented by one level to
      suit a file-scoped namespace; no statement was otherwise altered.

   c) Conditional compilation removed. The upstream source is guarded by three
      compile-time switches. They were evaluated once for this library's
      net10.0-only target and the dead branches, along with the #if/#else/#endif
      directives themselves, were removed:

          use_lines  ENABLED  upstream (open-path clipping retained)
          use_int32  disabled upstream (64-bit coordinates retained)
          use_xyz    disabled upstream (no Z member on IntPoint)

      Because use_int32 is off, the upstream "cInt" alias resolves to Int64 and
      was replaced throughout with the C# "long" keyword. This does not change
      any public signature.

   d) The upstream Int128 helper struct was REMOVED and replaced with the
      framework's System.Int128, which did not exist when the upstream code was
      written (it was introduced in .NET 7).

      The upstream struct was 178 lines, but the library used exactly one of its
      members: the static Int128Mul(long, long), at four sites in PolyClipBase
      (PointOnLineSegment and the three SlopesEqual overloads), and only inside
      equality comparisons of the form "Int128Mul(a, b) == Int128Mul(c, d)".
      Every other member of the struct - the +, -, unary -, <, >, != operators,
      IsNegative and the explicit double conversion - was unreachable from the
      library. Those four sites now read "(Int128)a * b == (Int128)c * d".

      This is the only modification that changes an expression inside a method
      body. It was verified by differential testing rather than by inspection: a
      verbatim copy of the upstream struct is retained in the test project (see
      DERIVED TEST SOURCE FILES above) and UpstreamInt128Tests asserts that both
      implementations reach an identical equality decision across every
      combination of 20 boundary values - 160,000 comparisons, including
      long.MinValue, where the upstream negation silently overflows - plus
      1,000,000 seeded random products drawn from the full coordinate range,
      with System.Int128 additionally cross-checked against System.Numerics.
      BigInteger as an independent oracle. No divergence exists.

      One upstream defect disappears as a consequence, and is recorded here
      because it was real: the upstream explicit double conversion negated with
      a one's complement (~lo + ~hi * 2^64) instead of a two's complement, so
      every negative value converted to a double one unit short in magnitude.
      That conversion was never reachable from the library.

   e) XML documentation comments were added to the public API surface. The
      upstream source carries almost no XML doc comments; the summaries were
      written for this port against the upstream HTML documentation shipped in
      clipper_ver6.4.2.zip (Documentation/Docs/Units/ClipperLib). These are
      additions only - no upstream comment was removed or reworded.

   f) Target framework. The upstream C# project targeted .NET Framework;
      this library targets net10.0 only.

   g) Syntax modernization. Changes were made in the CodeBrix.PolygonTools
      codebase to update the syntax of certain code areas to be aligned with
      .NET 10 and C# version 14 coding/syntax conventions - while preserving
      the original intentions of the code.

   Apart from the Int128 helper described in (d), no upstream file was omitted
   from the port, and no upstream type, method, property, field or enum member
   was dropped.
